## 主題一：怎麼下一個好的指令

### 壞的指令 vs 好的指令

```
❌ 壞：
「幫我修 bug」

✅ 好：
「UserService.java 的 login() 在密碼含特殊字元時回傳 500 錯誤，
 幫我找原因並修復，修完跑既有的 JUnit 測試確認沒有 regression」
```

差在哪裡？好的指令包含三個要素：

| 要素 | 壞的例子 | 好的例子 |
|------|---------|---------|
| **上下文** | 不知道是什麼 bug | 哪個檔案、哪個方法、什麼情況 |
| **範圍** | 不知道改哪裡 | 修 login()，跑測試 |
| **完成定義** | 不知道什麼叫做好 | 測試通過才算完成 |

### 幾個實用原則

**給夠的上下文，不要讓 Claude 猜：**
```
❌「這個函式有問題」
✅「src/utils/parser.js 的 parseDate() 在輸入 '2024-13-01' 時沒有拋出錯誤，應該要驗證月份範圍」
```

**說清楚你不想要什麼：**
```
「重構這個 class，但不要改動 public interface，其他測試不能壞掉」
```

**分步驟，不要一次丟一堆：**
```
步驟 1：先幫我看 AuthService.java，了解目前架構
步驟 2：（確認 Claude 理解後）再開始修改
```

**語言：** 用繁體中文對話完全沒問題，程式碼、檔名、指令保持英文。

## 主題二：對話中的基本操作

### @ 引用檔案

在對話框輸入 `@`，會跳出檔案選擇器，選完後 Claude 精準知道你在說哪個檔案：

```
@src/service/UserService.java 的 login() 方法有什麼問題？
```

比「那個 UserService 裡的登入方法」精準很多。也可以引用多個：

```
比較 @src/v1/Auth.java 和 @src/v2/Auth.java 的差異
```

### ESC — 中斷 Claude

Claude 做到一半發現方向不對？按 `ESC` 立刻中斷，比等它做完再 undo 省事。

中斷後重新描述，不要說「繼續但換個方式」——換方向就重新說清楚。

### Shift+Tab — 切換權限模式

在對話框按 `Shift+Tab`，可以在三種模式之間切換：

```
一般模式      → 每個操作都要你確認
auto-accept  → 自動接受所有操作（適合你信任的任務）
plan mode    → 只規劃不執行，適合複雜任務
```

詳細用法在 05 單元，這裡只需要知道這個快捷鍵存在。

### ! 執行 shell 指令

在訊息開頭加 `!`，直接執行 shell 指令，結果會進入對話上下文讓 Claude 看到：

```
! git log --oneline -10
! cat package.json
! npm test 2>&1 | tail -20
```

比「幫我跑 git log 然後分析」更快，也讓 Claude 看到真實輸出而不是猜。

### 貼上圖片

可以直接把截圖貼進終端機（macOS 用 `Cmd+V`），Claude 看得到畫面內容：

```
（貼上錯誤截圖）
這個錯誤訊息是什麼意思？怎麼修？
```

## 主題三：上下文管理

**新手最常忽略的問題：** 對話越長，Claude 越容易忘記前面的細節，回應品質會下降。

### 三個對話指令

**`/clear` — 開始全新任務**
```
/clear
```
完成一個任務、要開始完全無關的新任務時用。清空對話不是損失，是好習慣。

**`/compact` — 對話太長時壓縮**
```
/compact
```
保留重點摘要，讓 Claude 記得關鍵決策，但釋放 context 空間。感覺對話變慢或 Claude 開始重複問題時用。

**`claude --continue`（簡寫 `claude -c`）— 回到上一個對話**

在終端機執行：
```bash
claude --continue
```
回到你上次的對話，繼續做到一半的任務。

### 一個任務一個對話

```
✅ 好的習慣：
[新對話] 修復 login bug → /clear
[新對話] 重構 UserService → /clear
[新對話] 新增 API endpoint

❌ 壞的習慣：
[同一個超長對話] 修 bug、重構、新增功能、問架構問題...
（Claude 最後不知道在討論什麼）
```

## 主題四：跟 git 配合的安全工作流

### 開始大改動前

```bash
# 確認工作區乾淨
git status

# 如果有未 commit 的東西，先 commit 或 stash
git stash

# 複雜改動考慮開新分支
git checkout -b feature/xxx
```

**原因：** 改壞了隨時可以 `git checkout .` 還原，前提是工作區起點是乾淨的。

### 讓 Claude 改完後自己先看

```bash
# Claude 改完後，先看它動了什麼
git diff

# 確認沒問題再繼續
# 有問題就告訴 Claude 哪裡不對
```

不要無腦 accept 所有改動，看 `git diff` 是你的最後防線。

### 讓 Claude 寫 commit message

```
幫我 commit 目前的改動，根據這次做了什麼寫 commit message
```

Claude 會看 `git diff`，根據實際改動寫出合適的 message，比你手寫通常更完整。

### 改壞了怎麼還原

```bash
# 還原所有未 commit 的改動
git checkout .

# 還原特定檔案
git checkout src/service/UserService.java

# 還原到某個 commit
git reset --hard <commit-hash>
```

## 主題五：出錯時怎麼辦

### Claude 改錯方向

按 `ESC` 中斷，**重新描述**而不是讓它繼續猜：

```
❌「不對，繼續但換個方式」
✅ ESC → 「我要的是 X，不是 Y。具體來說：[重新說清楚]」
```

### 環境出問題

```
/doctor
```

Claude Code 自我診斷工具，能找出常見的設定問題。

### 常用查詢指令

```
/model    → 查目前用哪個模型，也可以切換
/help     → 所有可用的指令清單
```

## 練習

### 練習 A：@ 引用檔案

選你的專案裡任何一個你熟悉的檔案，用 `@` 引用它：

```
@[你的檔案路徑] 幫我解釋這個檔案的主要功能，以及有沒有你覺得可以改善的地方
```

確認 Claude 的回答是針對那個具體檔案的，不是泛泛而談。

### 練習 B：管理對話

開始一個對話，隨意聊幾個問題讓對話變長，然後：

1. 執行 `/compact`，觀察 Claude 如何摘要目前的對話重點
2. 再問一個問題，確認 Claude 還記得之前的脈絡
3. 執行 `/clear`，確認對話清空
4. 重新開一個新問題，感受「全新的 Claude」

### 練習 C：git 安全工作流

```bash
# 步驟 1：確認工作區乾淨
git status
```

叫 Claude 做一個小改動（例如：在任意檔案加一行註解）。

```bash
# 步驟 2：看它改了什麼
git diff
```

確認沒問題後：

```
幫我 commit 這個改動，用繁體中文寫 commit message
```

觀察 Claude 生成的 commit message，決定是否接受。

## 常見問題

**Q：中文指令 Claude 真的看得懂嗎？**
完全沒問題。Claude 理解繁體中文和英文一樣好，用你習慣的語言就好。唯一建議英文的是程式碼本身和檔案路徑，避免混淆。

**Q：@ 引用和直接說檔名有什麼差？**
直接說「UserService.java」Claude 要靠搜尋找到它；`@` 引用是你直接指定路徑，確保 Claude 看的是正確的檔案，尤其是同名檔案分散在不同目錄時差異很大。

**Q：/clear 會不會讓 Claude 忘掉 CLAUDE.md 的設定？**
不會，CLAUDE.md 是每次對話開始時自動載入的，/clear 之後開新對話一樣會讀取。

**Q：什麼時候用 /compact，什麼時候用 /clear？**
做同一個任務中途對話太長 → `/compact`（保留脈絡）。做完一個任務要開始新的 → `/clear`（全新開始）。

**Q：Claude 改了我不想要的東西，除了 git checkout 還有別的方法嗎？**
在 Claude 執行的當下可以按 `ESC` 中斷。如果已經改完，用 `git checkout` 是最乾淨的方式。Claude Code 沒有內建的「undo 上一個操作」功能，所以 git 工作區保持乾淨很重要。

## 完成條件

不看教材，能回答以下四個問題：

1. **怎麼中斷 Claude 正在做的事？** → 按 ESC
2. **怎麼讓 Claude 精準看到某個檔案？** → 輸入 @ 選取檔案
3. **做完一個任務要開始新任務，該做什麼？** → 執行 /clear
4. **Claude 改壞了怎麼還原？** → 確保工作區是乾淨的，git checkout 還原
